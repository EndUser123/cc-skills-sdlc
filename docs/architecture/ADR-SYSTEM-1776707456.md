# ADR: Browser Automation Architecture for yt-is

**Date:** 2026-04-20  
**Status:** Accepted  
**Deciders:** Bruce Thomson  

---

## Context

yt-is processes 140k+ YouTube videos for transcript extraction. The current browser automation layer (`csf_selenium.py`) uses a single persistent Firefox profile with GeckoDriver, fetching pages sequentially. Three candidate improvements were identified from competitive analysis:

1. **Playwright async context architecture** — Replace Selenium with Playwright for 2-5x bulk DOM speedup, native async, auto-waits
2. **Session browser + parallel page launch** — Same Firefox model but with parallel page contexts instead of sequential
3. **ytInitialPlayerResponse JSON parsing** — Bypass yt-dlp for live/premiere/no-captions by parsing embedded page JSON directly

---

## Decision

**Recommended order: Session browser + parallel launch → Playwright migration → ytInitialPlayerResponse**

### 1. Session Browser + Parallel Page Launch (Preferred First Step)

**Approach:** Refactor `csf_selenium.py` to use Firefox's context API instead of sequential page.goto() calls. A single Firefox profile can host multiple contexts, each with its own cookies and state. Within a single terminal, launch multiple pages in parallel within one context.

**Changes:**
- Replace sequential `driver.get(url)` loop with `context.new_page()` + `asyncio.gather(*(page.goto(url) for page in pages))`
- Use existing `BatchScheduler` concurrency model — no new multi-terminal infrastructure needed
- Firefox profile path stays the same; only the page dispatch model changes

**Tradeoffs:**
- ~2-3x throughput improvement over sequential (conservative estimate; Firefox context can handle 10-20 parallel pages)
- No new external dependencies
- Works within existing WAL safety model
- Failure: single page crash doesn't cascade (Firefox contexts isolate crashes)
- Risk: Firefox context parallel pages may hit YouTube's per-session rate limits faster than sequential

**Scope:** `csf/csf_selenium.py`

---

### 2. Playwright Migration (Medium-Term)

**Approach:** Replace `csf_selenium.py` Firefox/GeckoDriver with Playwright's `chromium.launch_persistent_context()`. Keep Firefox cookies for age-restricted content. Playwright's auto-wait and async API eliminate most flaky selector failures.

**Changes:**
- Add `playwright` to dependencies
- Replace `Selenium` driver lifecycle with Playwright browser/context/page lifecycle
- `chromium.launch_persistent_context(profile_dir=firefox_profile_path)` retains existing cookie persistence
- Async page dispatch same as pattern 1

**Tradeoffs:**
- ~2-5x bulk speedup (Playwright's async model, no WebDriver overhead)
- Auto-waits eliminate explicit `time.sleep()` retry loops
- Cross-browser: same API works for Firefox, Chrome, Webkit
- Cost: new dependency, Playwright browser installation (`playwright install chromium firefox`)
- Risk: Firefox-over-Chromium for age-restricted content — must verify cookie reuse works across Playwright Firefox contexts

**Scope:** `csf/csf_selenium.py`, `tests/test_csf_selenium.py` (new)

---

### 3. ytInitialPlayerResponse JSON Parsing (Lower Priority)

**Approach:** Before invoking yt-dlp or browser automation, attempt to extract `ytInitialPlayerResponse` and `ytInitialData` from the YouTube embed endpoint using `curl_cffi` (already in use by `fetchviaytdlp`). Parse the embedded JSON to extract transcript data for live/premiere/no-captions cases.

**Changes:**
- Add JSON extraction function in `transcript.py` — `curl_cffi` HEAD first, parse `ytInitialPlayerResponse` from response
- Falls back to existing yt-dlp chain if JSON extraction fails or returns no transcript
- No new dependencies (curl_cffi already present)

**Tradeoffs:**
- Eliminates yt-dlp overhead for a subset of videos
- Live streams and premieres often have transcripts embedded in page JSON — this reaches them directly
- Only useful if live/premiere/no-captions is a meaningful fraction of the backlog
- Risk: YouTube frequently changes page structure; JSON extraction is brittle without monitoring

**Scope:** `csf/transcript.py`

---

## Recommendation

**Start with Pattern 1 (Session browser + parallel launch).** It is:
- The lowest effort (no new dependencies)
- The lowest risk (same Firefox profile, same browser, same cookies)
- Immediately compatible with existing multi-terminal WAL model
- A prerequisite for Pattern 2 (the parallel dispatch architecture is identical; only the browser driver differs)

**Pattern 3 (ytInitialPlayerResponse) should be evaluated after measuring the live/premiere failure rate** in the current pipeline. If <5% of failures are live/premiere cases, the brittleness cost outweighs the gain.

---

## Consequences

### Happy Path
- Pattern 1 ships → 2-3x throughput improvement on DOM-heavy fetches
- Pattern 2 ships → 2-5x improvement, auto-waits, async API
- Pattern 3 ships → yt-dlp bypassed for live/premiere edge cases

### Failure Modes
- Pattern 1: YouTube per-session rate limit reached faster → add back-off delay between parallel batches
- Pattern 2: Cookie reuse breaks across Playwright Firefox context → fallback to Selenium for age-restricted
- Pattern 3: YouTube page restructure → JSON parsing silently returns empty, falls back to yt-dlp (safe degradation)

---

## Verification

- Pattern 1: Run `pytest tests/test_csf_selenium.py -v` with parallel dispatch disabled first (baseline), then enabled. Measure throughput delta on 100-video sample.
- Pattern 2: Run existing transcript test suite against Playwright Firefox context. Verify age-restricted video cookies persist.
- Pattern 3: Instrument `transcript.py` to count `ytInitialPlayerResponse` parse successes vs yt-dlp fallbacks on a 1000-video sample.
