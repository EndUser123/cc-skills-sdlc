# ADR: Browser Automation Architecture for yt-is

**Date:** 2026-04-20
**Status:** Draft — superseded by investigation findings
**Deciders:** Bruce Thomson

---

## Context

yt-is processes 140k+ YouTube videos for transcript extraction. The current browser automation layer (`csf_selenium.py`) uses a single persistent Firefox profile with GeckoDriver, fetching pages sequentially. Three candidate improvements were identified from competitive analysis:

1. **Playwright async context architecture** — Replace Selenium with Playwright for 2-5x bulk DOM speedup, native async, auto-waits
2. **Session browser + parallel page launch** — Same Firefox model but with parallel page contexts instead of sequential
3. **ytInitialPlayerResponse JSON parsing** — Bypass yt-dlp for live/premiere/no-captions by parsing embedded page JSON directly

## Corrections from Investigation (2026-04-20)

### Pattern 1 (Session Browser + Parallel Launch) — Architecture Correction

**Current state:** `csf_selenium.py` line 252 uses `ThreadPoolExecutor(max_workers=N)` — thread-level parallelism with one Firefox driver instance per worker thread. Within each worker, `driver.get(url)` is sequential. `transcript.py` line 1359 creates a **fresh** `webdriver.Firefox(...)` instance per call.

**Problem:** Firefox Selenium does not have a context/page parallelism API. The "context.new_page() + asyncio.gather" architecture is Playwright-specific. To get true parallel pages within a single browser session requires Playwright or Selenium Grid.

**Revised scope:** Pattern 1 as described is NOT a low-effort Selenium refactor. It is either a Playwright migration (Pattern 2) or a significant re-architecture.

### Pattern 3 (ytInitialPlayerResponse) — Value Is Narrower Than Modeled

**Current state:** yt-dlp internally parses `ytInitialPlayerResponse` and surfaces extracted subtitle URLs in `info_dict["automatic_captions"]`. yt-is then fetches subtitle JSON3 via curl_cffi. yt-dlp does NOT expose the raw `ytInitialPlayerResponse`.

**Corrected value:** Pattern 3 helps ONLY when yt-dlp fails entirely — live/premiere/no-captions where yt-dlp returns no subtitle URL. For those cases, parsing raw player_response JSON from the page would recover captions. For the common case (yt-dlp succeeds), Pattern 3 is redundant — yt-dlp already did the extraction.

**Implication:** Pattern 3's payoff is proportional to the `no_captions`/`live` failure rate in the NotebookLM path. If those are already being routed to NotebookLM and NotebookLM succeeds on most of them, Pattern 3 adds little. Measure before committing.

### Verification Infrastructure Gap

**Current state:** `tests/test_csf_selenium.py` is a smoke test (import + one assertion). No timing harness exists. `refactor_plan.md` line 167 is a TODO.

**Implication:** Throughput claims cannot be validated without a perf measurement harness.

### Routing Layer — NOT a Gap

`orchestrator.py` has failure-aware routing with per-channel history. No gap here.

## Revised Decision

**Revised order: Playwright migration first → ytInitialPlayerResponse (conditional)**

### 1. Playwright Migration

**Approach:** Replace `csf_selenium.py` Firefox/GeckoDriver with Playwright's `firefox.launch_persistent_context()`. Persistent context retains cookies for age-restricted content.

**Changes:**
- Add `playwright` to dependencies
- Replace `Selenium` driver lifecycle with Playwright browser/context/page lifecycle
- Async page dispatch: `context.new_page()` + `asyncio.gather(*(page.goto(url) for page in pages))`

**Tradeoffs:**
- ~2-5x bulk speedup (Playwright async, no WebDriver overhead)
- Auto-waits eliminate explicit retry loops
- True parallel pages within one browser context
- Cost: new dependency, `playwright install firefox chromium`
- Risk: Cookie reuse for age-restricted — fallback to Selenium for those specific videos

**Scope:** `csf/csf_selenium.py`, `tests/test_csf_selenium.py` (expanded)

---

### 2. ytInitialPlayerResponse JSON Parsing (Conditional)

**Pre-requisite:** Run yt-dlp on a 20-video sample of `no_captions` items that NotebookLM rejected. If ≥1 (5%) recover captions, proceed.

**Approach:** curl_cffi fetches YouTube page, parse `ytInitialPlayerResponse` for subtitle data. Falls back to `fetch_transcript_chain()` on failure.

**Tradeoffs:**
- Only valuable if the `no_captions`/`live` failure population has recoverable captions
- Brittle without monitoring — YouTube page structure changes break the parser
- Safe degradation: failures fall back to yt-dlp chain

**Scope:** `csf/transcript.py`

---

## Recommendation

1. Build timing harness (`tests/test_csf_selenium_perf.py`) — 100-video baseline before any ship
2. Migrate to Playwright — parallel dispatch + auto-waits + async in one change
3. Measure actual throughput delta
4. Conditional: run yt-dlp on 20 NotebookLM-rejected `no_captions` samples. If ≥5% recover captions → implement Pattern 3. Else drop it.

---

## Consequences

### Happy Path
- Playwright migration ships → parallel pages + async API + auto-waits + 2-5x speedup
- ytInitialPlayerResponse ships → yt-dlp bypassed for live/premiere edge cases (conditional)

### Failure Modes
- Playwright migration: Cookie reuse breaks → fallback to Selenium for age-restricted only
- ytInitialPlayerResponse: Page restructure → silent empty, falls back to yt-dlp (safe)
- Parallel pages: YouTube per-session rate limit hit faster → batch back-off delay

### Open Pre-requisites
- No perf measurement harness — must be built before verifying throughput claims
- ytInitialPlayerResponse value unmeasured — conditional on 20-video sample

---

## Verification

- Pre-requisite: timing harness, 100-video baseline
- Playwright: re-run timing harness, expect ~2-5x bulk speedup
- ytInitialPlayerResponse: 20-video sample, yt-dlp recovery rate ≥5% to proceed
