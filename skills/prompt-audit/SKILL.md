# /prompt-audit — Prompt Pattern Audit Skill

**Role:** Given a target skill, audit its SKILL.md and scripts against the [prompt-patterns-catalog](../prompt-patterns-catalog.md). Report pattern coverage as: **PRESENT**, **PARTIAL**, or **MISSING**.

**Catalog:** `P:/packages/cc-skills-sdlc/prompt-patterns-catalog.md` (P1–P8)

---

## What /prompt-audit Must Do

1. Resolve the target skill path
2. Read target SKILL.md + scripts directory
3. Check each P1–P8 pattern for presence
4. Report findings with specific evidence (file:line citations)
5. Emit a `prompt-audit_{skill}_{RUN_ID}.json` artifact

**What /prompt-audit Must NOT Do:**
- Modify the target skill
- Run verification commands against the target
- Replace a full code review

---

## Required Environment

```bash
export TERMINAL_ID="${TERMINAL_ID:-$(uuidgen | cut -d'-' -f1 | tr '[:upper:]' '[:lower:]')}"
export RUN_ID="${PROMPT_AUDIT_RUN_ID:-$(uuidgen)}"
export CATALOG="P:/packages/cc-skills-sdlc/prompt-patterns-catalog.md"
export SKILLS_ROOT="P:/packages/cc-skills-sdlc/skills"
mkdir -p ".claude/.artifacts/${TERMINAL_ID}/prompt-audit"
```

---

## Arguments

- `TARGET_SKILL` — name of the skill to audit (e.g. `go2`, `code`, `tldr-stats`)

---

## Audit Protocol

### STEP 1: Resolve target skill

```bash
TARGET_SKILL="$1"
SKILL_DIR="${SKILLS_ROOT}/${TARGET_SKILL}"
[ ! -d "$SKILL_DIR" ] && echo "ERROR: skill '$TARGET_SKILL' not found in $SKILLS_ROOT" && exit 1
STATE_DIR=".claude/.artifacts/${TERMINAL_ID}/prompt-audit"
mkdir -p "$STATE_DIR"
```

### STEP 2: Read target SKILL.md

Read the full SKILL.md. Note:
- Step sequences and their prose descriptions
- Script invocations (what scripts are called, with what env vars)
- Evidence requirement language (E1/E4 mentions, fact/inference framing)
- Checklist or review pass descriptions

### STEP 3: Read target scripts

For each `.py` file in `${SKILL_DIR}/scripts/`:
- Grep for P1–P8 tokens (pattern keywords, function names, output format strings)
- Note which patterns are found and in which file

### STEP 4: Pattern-by-pattern check

Run checks per pattern (see below). Collect **PRESENT / PARTIAL / MISSING** verdict per pattern.

### STEP 5: Write audit artifact

```bash
python -c "
import json, datetime
result = {
    'audit_run_id': '$RUN_ID',
    'target_skill': '$TARGET_SKILL',
    'audited_at': datetime.datetime.now(datetime.timezone.utc).isoformat(),
    'patterns': { ... }  # filled by step 4
}
path = '$STATE_DIR/prompt-audit_${TARGET_SKILL}_${RUN_ID}.json'
open(path, 'w').write(json.dumps(result, indent=2))
print(f'Audit written to {path}')
"
```

## Pattern Check Definitions

### P1 — [FACT]/[INFERENCE]/[RECOMMENDATION] Contract

**How to check:**
1. Grep scripts for `[FACT]` and `[INFERENCE]` tokens
2. Read `verify-task.py` output format — does it produce structured evidence blocks?

**Verdict:**
- **PRESENT** — scripts output `[FACT]/[INFERENCE]` blocks per command result
- **PARTIAL** — evidence blocks exist but only one tag used (e.g. only `[FACT]`)
- **MISSING** — no structured evidence format in any script output

---

### P2 — Adversarial Break-Case Enumeration

**How to check:**
1. Grep scripts for: `pure-plan-only`, `fake-plan-analytical`, `marker-camouflage`, `rationalale-camouflage`, `minimal-malformed-plan`
2. Check `review-passes.py` scope pass content

**Verdict:**
- **PRESENT** — scope pass includes all 5 break cases as explicit checklist items
- **PARTIAL** — 2–4 cases present but not all 5
- **MISSING** — no adversarial check in scope pass

---

### P3 — Gap → Opportunity 6-Dimension Scan

**How to check:**
1. Grep scripts for `scope_drift`, `scope_in`, or gap-detection logic
2. Check if any script compares `scope_in` against actual changed files
3. Check SKILL.md for any gap-scan step between task selection and verification

**Verdict:**
- **PRESENT** — gap drift check runs before or alongside verification commands
- **PARTIAL** — scope_in checked but only for existence, not 6-dimension analysis
- **MISSING** — no gap or scope drift detection

---

### P4 — Evidence Requirement Framing

**How to check:**
1. Read review pass files generated by `review-passes.py`
2. Check if findings sections require: file path + line number per finding
3. Check SKILL.md prose — does it say "cite file and line" anywhere?

**Verdict:**
- **PRESENT** — review findings explicitly require file path + line per finding
- **PARTIAL** — findings require evidence but no line-number constraint
- **MISSING** — no evidence citation requirement in review prose or output

---

### P5 — Forensic ADR Review Matrix

**How to check:**
1. Grep scripts for `matrix`, `adr`, `forensic`, or `implementation.*diff`
2. Check `pr-artifacts.py` output — does it generate a comparison table?
3. Check if review-passes.py has a design/planning-specific pass

**Verdict:**
- **PRESENT** — pr-artifacts generates an ADR-style matrix or review-passes has a design-specific pass
- **PARTIAL** — matrix referenced but not generated as structured output
- **MISSING** — no ADR-vs-reality comparison in any artifact

---

### P6 — Success Criteria Checklist (Priority-Ordered)

**How to check:**
1. Read target SKILL.md — search for numbered must-do lists
2. Check verification step prose — does it define failure conditions per criterion?
3. Grep for `success criteria`, `must do`, `before asking`

**Verdict:**
- **PRESENT** — SKILL.md verification step has explicit numbered criteria with failure conditions
- **PARTIAL** — numbered criteria exist but no failure conditions defined
- **MISSING** — no explicit success criteria in verification step prose

---

### P7 — Root Cause Tracing (F1–Fn + Inference Chain)

**How to check:**
1. Grep scripts for `F1`, `F2`, `root cause`, `causal`
2. Check if any script generates numbered factual findings before inference
3. Check review output for structured RCA format

**Verdict:**
- **PRESENT** — RCA/finding output uses F1/F2 numbering + inference chain
- **PARTIAL** — root cause language exists but numbering is absent or inconsistent
- **MISSING** — no structured RCA format in any output

---

### P8 — External LLM Judge (5-Axis JSON Output)

**How to check:**
1. Grep scripts for `judge`, `external`, `verdict`, `severity`, `failure_mode`
2. Check if any review step dispatches to an external LLM for semantic review
3. Check SKILL.md — does it mention an external reviewer subagent?

**Verdict:**
- **PRESENT** — skill explicitly dispatches to an external judge with 5-axis JSON output schema
- **PARTIAL** — external reviewer mentioned but no structured 5-axis schema
- **MISSING** — no external judge subagent in any step

## Prohibited Actions

- Modifying target skill files
- Running target skill's verification commands
- Generating findings or recommendations — only report coverage
